# Lessons

- When a room domain is filtered to include pre-shaded givens, backtracking cleanup cannot blindly reset those cells to `-1`; it must restore from `initialState` on branch unwind.
- Sto-Stone rigid-drop checks must validate every destination cell except self-overlap cells. Treating a drop as valid when only one destination cell is empty is incorrect.
- Sto-Stone acceptance cannot stop at per-column counts after the rigid drop. The final board must be exactly the contiguous bottom half of the grid, with no gaps above it.
- In the rigid-drop simulation, droppability cannot be cached across passes. A stone that was blocked can become movable after another stone falls, so the solver must recompute downward moves from the current board state each pass.
- `domainGen.py` is not part of the live solver path. Any future revival should start from the room-domain loop in `readPuzzle.py`, not from the current stale prototype.
- Lightweight CLI metadata commands should stay on a parse-only path. Building room domains and mutable solver state just to show puzzle counts adds unnecessary work and couples inspection to solver internals.
- If a legacy helper module must remain for compatibility, point it at the shared live implementation instead of carrying a second stale algorithm that can drift.
- Keep package `__init__` files lightweight during a module split. Pulling in high-level orchestration imports from `__init__` can create circular imports between generator, IO, and solver layers before the package finishes initializing.
- If the goal is a visibly cleaner architecture, moving implementation into packages is not enough by itself. The repo root should only keep true entrypoints and project metadata; legacy surfaces belong under an explicit `compat` package, not as top-level files.
- In this workspace, pytest's default temp/cache directories are unreliable on Windows/OneDrive. Prefer a repo-owned writable fixture directory for test outputs and disable cacheprovider if it causes permission noise.
- Windows subprocess-based CLI tests can fail under pytest with `WinError 6` unless stdin is detached. Use `stdin=subprocess.DEVNULL` in the test harness.
- A Sto-Stone uniqueness-count path cannot reuse successful leaf validation naively. `is_sto_stone(...)` leaves the dropped final board in place on success, so bounded multi-solution search must snapshot and restore the grid around successful Sto-Stone checks.
- For numbered-room generation, forcing room weights to sum to exactly half the board area dramatically improves candidate filtering. Any other total can never satisfy the final Sto-Stone bottom-half fill condition, so it is wasted search.
- When cloning a `Puzzle` through its legacy-dict surface, deep-copy the nested grids, domains, and drawn-stone lists first. Reusing those mutable references leaks solver-state mutations back into the original puzzle and can invalidate later uniqueness or duplicate checks.
- For unsolved generated puzzles, `write_puzpre(...)` cannot rely only on `state.drawn_stones`. If there is no solved state yet, it must fall back to `spec.initial_state` so pre-shaded givens survive round-trip.
- In generator metadata, `pre_solved_rooms` tracks fully revealed witness stones from the reveal policy, not rooms whose entire geometric area is pre-shaded.
- Once the old flat API is gone, clone live `Puzzle` objects directly instead of round-tripping through a dict bridge. Keeping a dead compatibility conversion path around just to support internal cloning adds maintenance surface without benefit.
- Reveal metadata should be derived from the actual revealed witness cells, not inferred from the reveal policy label. Once `few-cells` can reveal anywhere from 1 to 4 cells, fields like `revealed_cell_count`, `revealed_room_count`, and `pre_solved_rooms` need to come from the final revealed set.
- When a weighted reveal policy needs exact cutoff semantics, do not accumulate float thresholds directly. Use a decimal-safe path or another exact representation, or a boundary like `0.95` can slip into the wrong bucket because of float rounding.
- If a new generator export is meant to be public, update both `src/stostone/__init__.py` and the root `stostone/__init__.py` compatibility shim in the same change. Otherwise import behavior will diverge between package surfaces.
- When composing generator presets with explicit CLI filters, merge them by intersection. Minimum-style thresholds should keep the stricter maximum lower bound, and maximum-style thresholds should keep the stricter minimum upper bound.
- If preset names affect generation behavior, stamp those names into `GenerationResult` and puzzle metadata. Corpus files need to explain not just the outcome, but which preset family produced it.
- Corpus calibration should consume summary JSON from generation runs instead of invoking generation internally. That keeps expensive corpus creation explicit and lets the analyzer be deterministic, fast, and testable with synthetic summaries.
- Include board dimensions in generation summary payloads. Calibration needs a stable family key such as `4x4-4r`; relying only on output filenames leaves rejected-quality and duplicate records harder to analyze.
- Do not blindly apply combined calibration recommendations across board sizes. The first 4x4/6x6 corpus showed very different difficulty distributions, and the overall difficulty recommendation collapsed the hard band; that points toward size-aware presets rather than one global threshold change.
- When difficulty bands become size-aware, keep the global bands as the fallback and stamp the selected difficulty family/scale into generated metadata. Otherwise a generated puzzle can say `easy` without explaining which calibrated score range made it easy.
- Calibration corpus plans should resolve relative paths from the plan file, not the process working directory. That lets ignored artifact folders contain self-contained plans and outputs that can be rerun from either CLI or app code.
- When non-calibrated board families are allowed, use a deterministic heuristic fallback rather than the old global tiny-board bands. The metadata must identify whether the scale was `calibrated`, `heuristic`, or `global`.
- For larger generated boards, clue-carving and witness search need explicit budgets. Reject pathological room/domain shapes before constructive search, and record budget exhaustion in metadata instead of silently spending unbounded time.
- Treat zero-width calibration bands as a warning, not a preset to apply blindly. The first budgeted 8x8 corpus compressed scores into `hard 80-80`, which points to score-model saturation or insufficient variety before runtime bands should be changed.
- Difficulty score models must be versioned and reused by calibration. If summaries store old scores but still contain solve iterations and quality metrics, calibration should rescore them with the current model rather than preserving stale compression.
- When generation has already proven uniqueness, reuse the first-solution iteration collected during that proof for quality scoring. A separate post-generation solve duplicates work and can dominate larger-board corpus runtime.
- Large-board generation prefilters should stay conservative and sit before witness search only. Pair every new rejection metric with at least one known-searchable seeded layout test so optimization does not silently remove valid candidate families.
